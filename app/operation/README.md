# app/operation（操作生命周期与收敛运行时）

## 背景

在很多系统里，一个“写操作成功”并不等于“用户已经能看到最终结果”。

典型场景包括：

- HTTP 请求已经返回成功，但读模型/投影稍后才可见；
- 一个命令会影响多个域对象，需要等待多个子系统状态收敛；
- 调用方需要知道当前是“已接受”“处理中”“已完成”“已失败”，而不是只有一次性的 200/500；
- 前端或上游系统需要一种统一方式跟踪异步推进、处理超时、降级与重试。

现有框架中已经有：

- `process/saga`：偏补偿型编排；
- `process/workflow`：偏状态机推进；
- `process/lock`：偏串行化控制。

但仍缺少一层更靠近“应用操作”的中立抽象，用于回答下面这些问题：

- 一次操作何时算被系统接受？
- 一次操作何时算真正完成（settled）？
- 如果结果最终一致性延迟可见，框架如何统一跟踪？
- 如何把“操作状态”暴露给 HTTP / SSE / 前端轮询？
- 如何把幂等、收敛、可观测性纳入同一套生命周期模型？

`app/operation` 的目标，就是补上这一层。

## 目的

`app/operation` 旨在提供一套中立的“操作生命周期运行时”，统一表达：

- 操作被接受（accepted）
- 操作处理中（processing）
- 操作已收敛（settled）
- 操作失败（failed）
- 操作超时（timeout）
- 操作进入降级态（degraded）

它不是某个具体业务模块，也不要求操作一定是长任务；它服务的是：

- 同步接受、异步收敛的写接口；
- 最终一致性读模型可见性；
- 多阶段操作的状态公开与追踪；
- 前后端统一的操作跟踪协议。

## 核心定位

`app/operation` 不是下列概念的替代品：

- 不是 `workflow` 的别名；
- 不是 `saga` 的别名；
- 不是一个单纯的“后台 job 系统”；
- 不是只为 HTTP 前端服务的轮询工具。

它更接近于：

- 面向应用层的操作实例模型；
- 连接“命令提交”和“最终结果可见”之间空档的运行时；
- 对外暴露状态、对内协调收敛的统一外壳。

可以把关系理解为：

- `operation`：对外的操作生命周期；
- `workflow` / `saga`：可选的内部推进机制；
- `lock`：可选的串行化基础能力。

## 关键概念

### Operation

一次可跟踪的操作实例。通常包含：

- 标识（`operation_id`）
- 类型（如 `task.create`、`order.submit`）
- 发起者与目标资源
- 当前状态
- 结果、错误、元数据
- 创建/更新时间

### Status

统一的生命周期状态模型。建议至少支持：

- `accepted`
- `processing`
- `settled`
- `failed`
- `timeout`
- `degraded`

### Settlement

操作“何时真正完成”的判定机制。

它不应等同于“处理函数返回成功”，而应允许基于条件收敛，例如：

- 某聚合版本已持久化；
- 某投影已追到指定版本；
- 某读模型状态已达到目标谓词；
- 多个子条件全部满足。

### Idempotency

同一个外部请求重试时，不应重复执行副作用。

`app/operation` 应能与幂等键绑定，使调用方在重复提交时拿到同一操作实例或同一结果，而不是重复落库、重复发事件或重复补偿。

### Visibility

操作状态需要可见、可查询、可订阅。

这包括但不限于：

- 查询当前状态；
- 读取操作结果；
- 暴露阶段推进事件；
- 支持 SSE / 轮询 / 事件总线等不同上游消费方式。

## 功能点

`app/operation` 规划承载以下能力。

### 1. 操作模型

统一定义：

- 操作标识
- 操作类型
- 生命周期状态
- 结果与错误
- 元数据与可观测字段

### 2. 操作存储

提供最小化存储接口，用于：

- 创建操作实例
- 读取当前状态
- 推进状态
- 写入结果与错误
- 记录收敛时间与过期时间

存储实现应保持可替换，例如内存、SQL、缓存装饰器等。

### 3. 收敛协调

提供统一的收敛接口与策略模型，用于：

- 判断某操作是否已完成；
- 等待最终一致性可见；
- 组合多个收敛条件；
- 支持同步收敛与异步收敛两类模式。

### 4. 幂等支持

提供与 `Idempotency-Key` 或同类去重键对接的能力，用于：

- 检测重复提交；
- 返回既有操作实例；
- 避免副作用重复执行。

### 5. 状态公开

为上层适配器提供统一状态公开能力，例如：

- HTTP 查询状态；
- SSE 订阅状态推进；
- 轮询返回建议间隔；
- 面向前端/调用方的标准响应信封。

### 6. 可观测性

统一埋点与日志字段，例如：

- `operation_id`
- `operation_type`
- `trace_id`
- `initiator`
- `resource_id`
- `settle_latency`
- `timeout / degraded / failed` 计数

### 7. 影响范围传播（可选增强）

为上层应用提供“操作会影响哪些 scope / resource”的标准元数据出口，供前端或缓存层统一做失效刷新。

注意：框架只定义承载方式，不直接绑定具体业务 scope 命名。

## 非目标

`app/operation` 当前不直接负责：

- 编排复杂领域步骤定义（这是 `workflow` / `saga` 的职责）；
- 具体业务读模型实现；
- 前端 store 刷新逻辑；
- 特定传输层的强绑定实现；
- 特定数据库方言的存储细节。

## 与其他模块的关系

- `process/workflow`：可作为 operation 的内部推进机制。
- `process/saga`：可作为 operation 的内部补偿型推进机制。
- `process/lock`：可用于保证同一操作实例串行推进。
- `eventing`：可承载操作状态事件或收敛通知。
- `messaging`：可承载后台执行与异步回调。
- `httpx`：可用于公开标准操作响应与状态查询接口。

## 适用场景

适合接入 `app/operation` 的典型场景：

- 写接口返回成功，但读模型稍后才可见；
- 一个用户动作会同时影响多个模块，需要统一等待收敛；
- 前端需要统一处理“处理中 / 已完成 / 已降级 / 已失败”；
- 系统需要支持操作重试、幂等与状态追踪；
- 需要对外公开状态查询或状态流。

## 演进原则

- 先沉淀统一模型与接口，再补具体实现；
- 先保证最小可用，再逐步增强为可观察、可扩展、可替换；
- 优先保持中立命名与边界，不直接夹带具体业务语义；
- 不把“异步”“轮询”“HTTP”这些实现细节上升为核心概念。

## 初步分层建议

后续实现时，可考虑逐步拆分为：

- `app/operation`：模型与对外接口
- `app/operation/store`：持久化接口与实现
- `app/operation/settlement`：收敛条件与协调器
- `app/operation/idempotency`：幂等支持
- `app/operation/stream`：状态流/订阅抽象
- `app/operation/http`：HTTP/SSE 适配

本 README 当前先落设计背景、目的与功能点，作为后续实现的共识基线。
