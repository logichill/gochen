# 从 RBAC 走到分层授权：一步一步把问题讲清楚

几乎每个后端工程师都做过 RBAC。用户、角色、权限三张表，勾一勾、关联一下，权限就跑起来了。在很多年里，这套模型覆盖了 80% 的场景，也就没人想过再往前走一步。

然后有一天，你接手了一个平台化的系统：上面是平台方，下面是一堆租户，每个租户又可能分区域、分项目。你很自然地在原来的三张表上打补丁——加个 `tenant_id`、给角色起个带前缀的名字、在中间件里塞点判断——然后你会发现，每次再有新业务进来都得再打一个补丁。补丁摞补丁之后，没人再说得清"管理员"到底是什么意思。

这篇文章不是要推销某个框架，而是想把"RBAC 到底在哪些地方不够用"拆开来一点点看。每往前走一步，都是被一个真实的场景逼出来的。等你走完这条路，回头再看就会明白：最终那套看上去有点复杂的模型，其实是被问题**逼出来**的，不是设计者刻意为之。

## 1. RBAC 在简单世界里是优雅的

先为 RBAC 说句公道话。

它很薄、很好理解。核心就三个概念：

- **User**：谁在操作
- **Role**：一组权限的名字，只是为了方便批量分配
- **Permission**：允许做什么动作

关系也很直白——用户被分配角色，角色里打包了权限，所以用户就拥有了这些权限。

这套模型能跑得很稳，是因为它背后有一个**暗含前提**：整个系统只有**一层**管理边界。大家都在同一间办公室里，"admin" 就是那个能管所有东西的人，"user" 就是普通用户，不会有歧义。

只要这个前提成立，RBAC 就够用。代码里判断一下 `hasPermission(user, "user.edit")`，事情就结束了。

但当你开始做平台化系统，这个前提就不成立了。

## 2. 第一个裂缝：同一个 "admin"，含义已经不一样了

想象一下这个场景：

- 有一位平台运营同学，她是整个平台的管理员，能看到所有租户的数据
- 有一位客户方的 IT 负责人，他是自己公司这个租户的管理员，只能看自己租户内部的数据

这两个人，在权限表里很自然都会被叫作 "admin"。可是你心里清楚，他们不是同一种 admin。

如果你还用经典 RBAC，硬要表达这件事，只能给角色换个名字——`platform_admin`、`tenant_admin`，然后再定义新角色 `region_admin`、`project_admin`……很快你会发现，角色表不再是"一组权限的名字"，而变成了"权限 × 范围"的笛卡尔积。每接一个新范围，你就得把所有角色复制一遍。

问题出在哪儿？

**RBAC 能回答"你有没有这个权限"，但回答不了"这个权限在什么范围里生效"。**

所以，我们需要的不是更多角色，而是让角色和"范围"解耦。最小的一步补丁是，把原来那句话：

> 用户 **有** 某个角色

改成：

> 用户 **在某个范围里** 有某个角色

为了承载这句话，我们引入第一个新对象：**UserRoleBinding**。

它是一条正式的授权关系记录，不是新的角色，也不是新的权限，只是把"用户和角色的关系"从一条边升级成一张表，额外记录了"这个角色在哪里生效"：

```
UserRoleBinding: (user_id, role_id, grant_scope_id)
```

于是同一个用户可以同时是 `admin @ platform` 和 `admin @ tenant:default`——两条不同的 binding，同一个角色模板，在不同的边界里生效。

角色表重新回到了"一组权限的名字"这个朴素定义，不用再爆炸。

## 3. 那"范围"到底是什么

上一节我们顺嘴用了一个词：**范围**。但它具体是什么？

它不能是一个字符串标签，因为平台要能"覆盖"租户——你需要表达一棵树。所以很自然，"范围"得是一等对象：**Scope**。

```
platform
├── tenant:A
│   ├── region:east
│   └── region:west
├── tenant:B
└── tenant:C
```

Scope 是授权意义上的"辖区"，承载三件事：

- 我的角色在哪个边界里生效
- 这个边界能不能覆盖下一级边界
- 这次查询应该看到哪个边界下的数据

这时候很多人会问一个合理的问题：

> 我们不是已经有 `tenant` 了吗？直接拿 tenant 当范围不行吗？

不行。这是一个很值得停下来想想的地方。

**Tenant 是业务概念**：它回答的是"这是谁的业务空间"、"这一单账单算谁的"、"这些配置归谁维护"。它关心的是**业务隔离**。

**Scope 是授权概念**：它回答的是"谁能覆盖谁"、"这个角色在哪生效"。它关心的是**管理覆盖关系**。

你可能会觉得这两个概念在九成场景下是重合的——每个 tenant 就对应一个授权范围，`platform` 就是所有 tenant 之上的根。确实如此，但关键在于**剩下那一成**。当你给 tenant 挂一个 "平台租户" 来表达 `platform` 时：

- 它没有业务意义（平台不是一个租户，它没有账单）
- 但它占着一个 tenant_id，其他所有业务代码都得想想"这条到底算不算数"

更稳的做法是让它们保持独立：tenant 做业务隔离，scope 做授权范围，tenant 通过一个 `root_scope_id` 挂到 scope 树上。这样 `platform` 就是一个干净的 scope，不需要任何"平台租户"这种语义污染。

到这里，我们已经能表达"一个用户在某个范围里有某个角色"。但这只是权限决策的**主体**侧。事情的另一半——那条被操作的数据——我们还没碰过。

## 4. 第二个裂缝：平台管理员为什么看不到自己管的数据？

我们来做一个很诚实的查询。

假设你在用户表里加了一个 `scope` 字段，标记这条用户数据属于哪个范围。平台管理员登录进来，想看一下全平台的用户，于是你在查询里加一句：

```sql
where user.scope = :current_user_scope
```

这时候平台管理员的 `current_user_scope = 'platform'`，而所有用户数据的 scope 是 `tenant:A`、`tenant:B`……**一个都匹配不上**。平台管理员在自己管的平台上，看到了一张空表。

很多团队到这里会往代码里撒一把胶水：

```python
if user.is_platform_admin:
    # 平台管理员放行所有数据
    return all_users()
else:
    return users_where(scope == user.scope)
```

这行 `if` 看上去没什么。但第二天产品说 "region 管理员要能看 region 下所有 project 的数据"，第三天又说 "某种特殊角色也要跨租户看"。这个 `if` 会越长越长，最后没人敢动。

问题的根源不是 `if` 写得不够好，而是——**等值过滤这个模型本身就不适合分层授权**。

等值只能表达"同一层的隔离"：A 看 A 的数据，B 看 B 的数据。但分层授权的本质是**上级覆盖下级**：平台能看所有租户、租户能看自己下面的 region、region 能看自己下面的 project。这是一个**子树**关系，不是一个**相等**关系。

换个角度想一下。"平台管理员应该看到哪些数据？"——答案是"平台这棵子树下所有 scope 的数据"。"租户 A 的管理员应该看到哪些数据？"——答案是"tenant:A 这棵子树下所有 scope 的数据"。

所以我们真正要算的不是"当前用户的 scope 等于什么"，而是：

> 当前用户**能够看到**的 scope 集合是什么？

一旦把问题改成这个形式，答案就清楚了：它是一个**集合**，不是一个**值**。查询不再是 `scope = x`，而是 `scope IN (x1, x2, x3, ...)`。

我们给这个集合起个名字：**DataScope**——当前会话默认能看到哪些 scope 下的数据。

```
DataScope {
    ActiveScope:     当前以哪个身份工作
    VisibleScopes:   从这个身份出发，能看到哪些 scope 的数据
}
```

有了这个东西，repo 层就不再需要写业务相关的 `if`——它只做一件统一的事：

```sql
where resource.<scope_field> IN :data_scope.visible_scopes
```

平台管理员的 `VisibleScopes` 是整棵树，所以能看见所有；租户管理员的 `VisibleScopes` 是自己那一棵子树，所以只能看见自己。同一句 SQL，不同的参数，不再需要特判。

## 5. 资源归属必须是显式的

上一节我一直在说"资源的 scope"，好像这个字段天然就在那儿。但仔细想想，它**从哪来**？

很多系统里是这样的：资源表有一个 `tenant_id`，大家顺手就拿它当权限过滤条件。看上去没问题，但 `tenant_id` 这个字段其实同时承担了两件事：

1. **业务隔离**：这条账单算谁的、这笔消费归谁的统计
2. **数据权限边界**：谁有资格看到这条数据

这两件事绝大多数时候是一致的，但它们**原因不同**。一致是巧合，不是因果。

考虑一个反例：一张菜单数据，是平台配置的（所有租户都看同一份），你会给它 `tenant_id = ?` 吗？填哪个 tenant 都不对。填 null 又让业务逻辑里到处判空。

或者反过来：某个租户内部开了几个区域，区域之间要互相隔离，但都属于同一个 tenant。这时候 `tenant_id` 就不够细了，你还得再加一个 `region_id`，然后权限代码又得分别处理这两个字段。

把两件事用一个字段表达，一开始省事，后面每次需求变化都得动这个字段，而它的每一个变化都既影响业务又影响权限。

所以更稳的做法是给每个受控资源加一个独立字段：

```
managed_scope_id  —— 这条数据归哪个管理范围负责
```

它**只**表达权限归属。业务隔离该用 `tenant_id` 还用 `tenant_id`，两者互不干扰。菜单的 `managed_scope_id = platform`，没有 tenant_id；某个区域的私有数据 `managed_scope_id = region:xxx`，同时 `tenant_id` 还是那个租户。职责分开之后，每个字段都可以独立演化。

还有一个值得注意的约束：**`managed_scope_id` 一旦写入就不应该随便改**。"把这条数据从一个范围迁到另一个范围"是一个高危动作，它直接改变了谁能看谁能改，不能藏在普通的 update 里顺手完成。更健康的做法是把它做成一个显式的 `reassign` 操作，走独立的权限检查和审计。

## 6. 第三个裂缝：同一个人登录进来，到底是谁？

再回来看主体侧。

我们前面说过，同一个用户可以同时在多个范围里有多个角色。于是问题来了：**他这次登录，到底算以哪个身份在工作？**

- 按 role binding 里第一条？随机。
- 按某个特殊角色名（比如 `system_admin`）？换个叫法就翻车。
- 按 tenant 是不是 platform？回到"用 tenant 兼职 scope"的老路。

所有这些"猜"的方案都会在某一天咬你一口。用户切换了账号身份，看到的菜单对不上；或者明明刚刚还能操作的资源，刷新一下变成 403。调试这种问题比写它痛苦得多。

正确的做法其实很朴素：**别猜，让用户自己说**。

把登录拆成两步：

1. **Authenticate**——验证用户名密码，返回这个人**可用**的身份列表（一堆 scope / binding 选项）
2. **ActivateScope**——用户在这些选项里**显式选**一个，告诉系统"这次我以这个身份工作"

之后签发的 token 里就带着这一次的 `active_scope_id`。每一个请求的权限判定，都以这个字段为准。如果用户想切换身份，就重新激活一次。

这一步不新增概念，只是把"我现在是谁"从**隐式推断**变成了**显式状态**。它看起来很小，但它是整套模型能稳定跑起来的**锚点**——一旦 active scope 稳了，下游所有判定都稳了。

## 7. 把线穿起来

到这里，每个概念都有了，一次请求的完整路径其实已经串起来了：

```
1. 登录时选定 ActiveScope，之后每次请求都带着它
2. 根据 ActiveScope 计算 DataScope——当前能看到哪些 scope 的数据
3. 查询时 repo 自动注入：where managed_scope_id IN data_scope.visible_scopes
4. 写入时先做权限判定，通过后把判定结果转成 WriteConstraint（一个"允许写哪些 scope 下哪些资源"的显式约束），repo 据此落库
```

查询侧讲完了，写入侧我多说一句。

很多人会觉得权限判定通过了就行，剩下交给 service 去写数据。但**判定通过的那一刻**和**真正落库的那一刻**之间，上下文可能已经变了——可能是并发的另一个请求改了数据，可能是 service 自己手抖传了个错误的 id。

更稳的做法是让"判定结果"变成一个可以传递的对象，它同时告诉 repo："你被授权写的，是这几条资源、这几个 scope、这些 revision"。repo 在最终 update 的 where 里把这些全部加上。如果 `RowsAffected = 0`，区分是"目标不存在"、"版本冲突"、还是"越权导致没命中"，而不是静默成功。

这个"可传递的授权结果"就是 **WriteConstraint**。它其实不是一个新概念，只是把"判定"和"落库"这两个时间点之间的默契**显式化**而已。

## 8. 性能：VisibleScopes 不能每次动态算

把理论说完了，来聊一个现实问题。

`VisibleScopes = 当前 scope 这棵子树下的所有后代 scope`。你可以每次请求都递归一遍 scope 树算出来，但很快就会发现：

- scope 树可能很深（平台 → 租户 → 区域 → 项目 → ……）
- 每次请求都算一次，大量重复
- 递归本身对数据库不友好

但反过来想，**scope 树本身变化频率极低**（新增一个租户、新增一个区域都不是每秒发生的事），而查询频率极高。这是一个典型的"写少读多"场景。

经典解法：**预展开映射表**。

维护一张 `scope_visibility_map`：

```
viewer_scope_id | target_scope_id
----------------|----------------
platform        | platform
platform        | tenant:A
platform        | region:A.east
platform        | tenant:B
tenant:A        | tenant:A
tenant:A        | region:A.east
tenant:A        | region:A.west
...
```

每一行都回答一个问题："viewer 这个范围能不能看到 target 这个范围的数据？" 能看到就有这一行，看不到就没有。

scope 树变化的时候（新增/删除节点），在同一个事务里同步刷新这张映射表。查询的时候不再递归，直接 join：

```sql
select r.*
from resource r
join scope_visibility_map svm
  on svm.target_scope_id = r.managed_scope_id
 and svm.viewer_scope_id = :active_scope_id
```

或者先一把捞出 viewer 的可见集合，然后 `managed_scope_id IN (...)`。两种写法都能走索引，都是 O(log n) 的查询。

这招不新，本质就是把一个图遍历问题变成一个索引查找问题，用写入时的小成本换查询时的大收益。

## 9. 回顾这条路

回头看我们走过的每一步：

| 问题 | 引入的概念 |
|---|---|
| 同一个角色名，在不同地方含义不一样 | **UserRoleBinding**：角色在哪个范围里生效 |
| "范围"本身需要被系统理解和覆盖 | **Scope**：独立于 tenant 的授权范围树 |
| 平台管理员用等值过滤看不到下级数据 | **DataScope**：当前能看到的 scope 集合 |
| 资源的"业务归属"和"权限归属"耦合在一起 | **managed_scope_id**：独立的权限归属字段 |
| 多身份用户登录后"我是谁"不明确 | **ActiveScope**：显式选择而不是推断 |
| 授权判定和落库之间的默契需要显式化 | **WriteConstraint**：可传递的授权结果 |
| VisibleScopes 动态计算太贵 | **scope_visibility_map**：预展开的可见性映射 |

每一步都是被一个具体场景逼出来的。这条路从头到尾没有跳跃——你可以在任何一步停下来反问"这个问题我真的遇到了吗？"如果没遇到，那一步对你就还不必要。RBAC 不是"过时的"，分层授权也不是"更先进的"。它们只是在回答不同的问题。

## 10. 一句话收尾

RBAC 回答的是：**你能做什么？**

分层授权回答的是：**你在哪里、以谁的身份、对哪些数据、能做什么？**

多出来的那几个限定词，每一个都对应着一个真实世界里的复杂度。不是设计者想装复杂，是世界本来就这么复杂。
